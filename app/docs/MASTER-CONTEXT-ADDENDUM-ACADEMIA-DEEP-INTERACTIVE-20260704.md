# Master context addendum - Academia profunda e interactiva

Fecha: 2026-07-04

## Proposito

Este addendum complementa el documento maestro de continuidad CXOrbia TyA y debe cargarse en Fuentes.

Su objetivo es evitar que Academia quede como texto superficial. Academia debe convertirse en un modulo profundo, editable, interactivo, accionable y alineado con cada modulo real de la plataforma.

## Regla principal

Academia no debe ser solo descripcion. Debe ser una experiencia de aprendizaje operativa y comercial dentro de la plataforma.

Cada curso, manual, leccion o guia debe explicar:

- que es;
- para que sirve;
- por que importa;
- que valor agrega;
- que problema resuelve;
- que rol lo usa;
- que flujo operativo sigue;
- donde se hace clic;
- que boton se usa;
- que datos se ingresan;
- que validacion confirma que quedo bien;
- que errores frecuentes ocurren;
- que hacer si algo falla;
- que beneficio comercial o de venta cruzada existe si aplica.

## Requisito de profundidad

Claude tiende a resolver superficialmente. Por eso, en cada auditoria se debe verificar que Academia tenga profundidad real.

No basta con textos como:

- "este modulo sirve para gestionar visitas";
- "aqui puedes ver reportes";
- "este curso explica postulaciones".

Debe existir contenido funcional como:

- pasos concretos;
- flujos completos;
- decisiones segun rol;
- ejemplos practicos;
- checklists;
- errores frecuentes;
- estados esperados;
- consecuencias operativas;
- valor para el negocio;
- elementos interactivos.

## Formato esperado en plataforma

Academia debe tener diseno de plataforma, no solo parrafos.

Debe incluir, segun aplique:

- tarjetas de rutas por rol;
- cursos por modulo;
- lecciones paso a paso;
- acordeones por tema;
- checklists interactivos;
- progreso por usuario;
- etiquetas de nivel: basico, operativo, avanzado, comercial;
- filtros por rol, modulo, proyecto, pais o paquete;
- busqueda;
- glosario clickeable;
- preguntas frecuentes;
- ejemplos de casos reales o simulados;
- botones para solicitar capacitacion;
- notificaciones de cambios o nuevas lecciones;
- relacion con novedades y actualizaciones de producto;
- acceso a manuales descargables o consultables;
- recursos adjuntos cuando existan.

## Edicion inteligente

Academia debe poder administrarse desde la plataforma.

Debe existir o planearse una ficha inteligente para crear y editar:

- cursos;
- lecciones;
- manuales;
- rutas de aprendizaje;
- glosario;
- preguntas frecuentes;
- checklists;
- temas de capacitacion;
- recursos adjuntos;
- paquetes o modulos comerciales.

Cada ficha debe permitir:

- crear;
- editar;
- borrar o archivar;
- duplicar;
- versionar;
- asignar rol objetivo;
- asociar modulo;
- asociar proyecto o tenant si aplica;
- adjuntar recursos;
- crear contenido desde texto;
- crear contenido desde recurso adjunto;
- marcar contenido como borrador, en revision, publicado, archivado;
- registrar ultima actualizacion;
- notificar cambios a roles afectados.

## Generacion asistida

La plataforma puede preparar borradores de contenido a partir de texto o recursos adjuntos, pero nunca debe publicar automaticamente contenido generado por IA sin revision humana.

Estados sugeridos:

- draft;
- ai_draft;
- human_review_required;
- approved;
- published;
- archived.

Gemini puede apoyar en Phase B para estructurar borradores, preguntas, checklists y glosarios, siempre con revision humana.

## Acceso en Top bar

Por la importancia de Academia, debe evaluarse y documentarse como pendiente de prototipo que Academia tenga acceso directo en el Top bar o en una ubicacion persistente equivalente.

El acceso debe permitir:

- entrar a mi ruta de aprendizaje;
- ver cursos pendientes;
- ver notificaciones de actualizaciones;
- solicitar capacitacion;
- buscar ayuda por modulo;
- consultar manuales;
- ver novedades de producto y lecciones nuevas.

## Notificaciones propias de Academia

Academia debe tener notificaciones propias o integradas al sistema de novedades.

Debe notificar:

- curso nuevo asignado;
- leccion actualizada;
- manual actualizado por cambio de modulo;
- nueva version de curso;
- solicitud de capacitacion recibida;
- capacitacion programada;
- capacitacion completada;
- contenido pendiente de revision humana;
- paquete o modulo nuevo disponible;
- cambio relevante en flujo operativo.

## Auditoria obligatoria para Claude

Cada vez que Claude entregue un paquete, se debe auditar si Academia fue tratada superficialmente.

Checklist minimo:

1. Que modulos cambiaron.
2. Que cursos/manuales relacionados se actualizaron.
3. Si el contenido esta por rol.
4. Si tiene profundidad operativa.
5. Si explica botones, pasos, estados y errores.
6. Si incluye valor agregado y beneficios.
7. Si tiene formato de plataforma e interactividad.
8. Si es editable desde fichas inteligentes.
9. Si permite borrar/archivar/duplicar/versionar.
10. Si tiene notificaciones o pendientes de notificacion.
11. Si el Top bar o acceso persistente fue considerado.
12. Si se documentaron pendientes cuando algo no se implemento.

Si Claude entrega solo textos superficiales, debe devolverse como pendiente, no aceptarse como resuelto.

## Relacion con cambios de modulos

Cada vez que se cambie un modulo, se debe revisar si se actualizan:

- manual del modulo;
- curso relacionado;
- leccion paso a paso;
- guia de botones;
- checklist de validacion;
- errores frecuentes;
- glosario;
- ruta por rol;
- notificacion de cambio;
- tema para solicitar capacitacion.

## Requisito por roles

Academia debe mantenerse segmentada por rol:

- Superadmin;
- Admin;
- Operativo/coordinador;
- Shopper;
- Cliente final;
- Consultora, representante, franquiciado, aliado o socio.

Cada rol debe ver lo que necesita aprender para operar y aprovechar la plataforma sin recibir contenido innecesario o confuso.

## Valor comercial

Academia tambien debe explicar beneficios de:

- modulos actuales;
- modulos premium;
- paquetes adicionales;
- automatizaciones;
- integraciones;
- servicios nuevos;
- soporte y capacitacion;
- uso avanzado de reportes.

Esto ayuda a adopcion, venta cruzada y retencion.
